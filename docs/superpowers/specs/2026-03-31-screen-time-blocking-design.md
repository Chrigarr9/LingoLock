# Screen Time App Blocking — Design Spec

## Overview

LingoLock uses Apple's Screen Time API (Family Controls framework) to block distracting apps at the system level. Users must complete FSRS flashcard reviews to unlock blocked apps for a 10-minute window. Escalation is exponential until all due cards are cleared, then flat. This replaces the existing Shortcuts-based locking mechanism entirely.

## Core Feature Loop

1. User opens any blocked app -> iOS shows a branded LingoLock shield
2. Shield displays: LingoLock icon, "Complete vocabulary in LingoLock", button "Open LingoLock"
3. User taps button -> ShieldActionExtension posts a local notification with deep link (`lingolock://challenge?source=screentime`) and returns `.close`
4. User taps notification -> LingoLock opens to challenge screen
5. Main app calculates required card count (escalation logic + FSRS due count)
6. User completes required cards -> main app clears shields via `ManagedSettingsStore` and starts a 10-minute `DeviceActivityMonitor` schedule
7. After 10 minutes, `DeviceActivityMonitorExtension.intervalDidEnd` fires -> re-applies shields

## Escalation Logic

- Unlock 1: 3 cards
- Unlock 2: 6 cards
- Unlock 3: 12 cards
- Unlock N: `3 * 2^(N-1)` cards
- **Cap:** Once all FSRS due cards are completed for the day, cost drops to flat 3 cards per unlock for the rest of the day
- All counters (unlock count, due-cards-cleared flag) reset at midnight
- Scope: **global** — one escalation counter, one unlock timer, completing cards unlocks all blocked apps

## No Bypass

There is no skip, emergency unlock, or cooldown bypass. Users who need to access a blocked app urgently can:

- Disable App Blocking in LingoLock settings
- Revoke Screen Time authorization in iOS Settings > Screen Time

Both options introduce enough friction to prevent impulsive bypassing.

## Setup & Configuration UI

### Onboarding Flow

1. "App Blocking" section in LingoLock settings
2. User taps "Set Up App Blocking"
3. LingoLock requests Individual Authorization via `AuthorizationCenter.shared.requestAuthorization(.individual)`
4. iOS shows system consent dialog
5. On approval, `FamilyActivityPicker` (Apple's standard app selection UI) appears. Users can pick individual apps, entire categories, or web domains. The picker is Apple's native UI — LingoLock does not control its layout or filtering.
6. User confirms selection -> shields applied immediately

### Settings Screen

- Toggle: App Blocking on/off (removes/re-applies all shields)
- "Blocked Apps": re-opens `FamilyActivityPicker` to edit selection
- Unlock duration: fixed at 10 minutes (displayed, not configurable)
- Status line: "Today: 3 unlocks, 21 cards completed"

### Technical Note

`FamilyActivityPicker` returns opaque `ApplicationToken` sets (privacy-preserving — no app names or bundle IDs visible to LingoLock). Tokens are stored in App Group `UserDefaults` for extension access.

## Architecture

### Principle: Dumb Extensions, Smart Main App

All intelligence (FSRS due count, escalation calculation, card counting) lives in the main React Native app where MMKV and the FSRS scheduler are available. Extensions are minimal — they block, redirect, and re-apply shields. Nothing more.

### Extension Targets

Three iOS extension targets generated by `@kingstinct/expo-apple-targets`:

| Extension | Purpose | Key Behavior |
|---|---|---|
| ShieldConfigurationExtension | Customizes shield appearance | Static branded message: icon, title, subtitle, button label |
| ShieldActionExtension | Handles button taps on shield | Posts local notification with deep link, returns `.close` |
| DeviceActivityMonitorExtension | Re-applies shields after unlock expires | `intervalDidEnd` -> reads tokens from App Group -> applies shields |

### Shared State (App Group UserDefaults)

Only two keys shared between main app and extensions:

| Key | Type | Purpose |
|---|---|---|
| `blockedAppTokens` | Data | Encoded `Set<ApplicationToken>` — which apps to shield |
| `blockingEnabled` | Bool | Master toggle |

All other state (escalation count, due card count, unlock history) stays in MMKV within the main app.

### Data Flow

```
Main App (React Native)
  |-- FamilyActivityPicker -> stores tokens in App Group
  |-- On challenge complete:
  |     |-- Clears ManagedSettingsStore shields
  |     |-- Starts 10-min DeviceActivityMonitor schedule
  |-- On foreground: checks if unlock expired, re-applies if needed
  |-- Tracks escalation count + due-cards-cleared flag in MMKV

ShieldConfigurationExtension
  |-- Renders static branded shield (no dynamic data needed)

ShieldActionExtension
  |-- Button tap -> posts local notification -> returns .close

DeviceActivityMonitorExtension
  |-- intervalDidEnd -> reads blockedAppTokens from App Group
  |-- Re-applies shields via ManagedSettingsStore
```

## Shield-to-App Redirect

Apple's `ShieldAction` does not support `.openParentApp`. The redirect uses a local notification:

1. ShieldActionExtension posts notification: "Tap to start your vocabulary challenge"
2. Notification carries deep link: `lingolock://challenge?source=screentime`
3. ShieldActionExtension returns `.close` (user goes to home screen)
4. User taps notification -> LingoLock opens to challenge screen

If the user dismisses the notification, the shield re-appears on next blocked app open. The notification persists in Notification Center.

LingoLock already requests notification permission for FSRS reminders. If not yet granted, the App Blocking setup flow requests it.

## Shortcuts Removal

The existing Shortcuts-based locking is removed entirely:

- Remove `PracticeNeededIntent` and `StartPracticeIntent` App Intents
- Remove grace period / cooldown logic from UserDefaults
- Remove Shortcuts setup instructions from the app
- Clean break — Screen Time API is the sole blocking mechanism

## Library

Uses `react-native-device-activity` by Kingstinct (v0.6.1):

- Peer dependencies: Expo >=52, React >=18.3, React Native >=0.76
- Covers: FamilyActivityPicker, ManagedSettingsStore, ShieldConfiguration, ShieldAction, DeviceActivityMonitor
- Generates extension targets via `@kingstinct/expo-apple-targets`
- Pre-1.0 but actively maintained (~145 GitHub stars)

## Entitlement Strategy

- **Development/testing:** Family Controls entitlement works locally without Apple approval
- **Distribution:** Requires `com.apple.developer.family-controls` restricted entitlement
- **When to apply:** After LingoLock is live on App Store
- **Framing:** Educational/parental tool — "LingoLock helps users and parents ensure language practice is completed before accessing entertainment apps"
- **If rejected:** Feature does not appear in the app; all other functionality unaffected
- **No feature flag needed:** App Blocking settings section is only visible when Screen Time authorization API is available (physical iOS device with approved entitlement)

## Decisions Summary

| Decision | Choice |
|---|---|
| Setup | Preset bundles + custom selection via FamilyActivityPicker |
| Escalation | Exponential (3 x 2^(N-1)) until due cards cleared, then flat 3 |
| Unlock window | Fixed 10 minutes |
| Scope | Global — one counter, unlocks all blocked apps |
| Initial free period | None — cards required from first open |
| Bypass | None — disable in settings if needed |
| Shield redirect | Local notification with deep link |
| Architecture | Dumb extensions, smart main app, minimal shared state |
| Shortcuts | Removed entirely |
