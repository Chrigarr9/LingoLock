## 2. Storage Platform Adapter

### Problem
`react-native-mmkv` is a native C++ module — it will NOT work on web. All storage flows through `src/services/storage.ts` which imports `createMMKV` directly.

### Solution: Platform-Specific Files
React Native's module resolution supports `.web.ts` suffix:
- `storage.ts` → used on native (keeps MMKV)
- `storage.web.ts` → used on web (uses localStorage)

Both files export the **exact same API** (same function signatures):
- `saveCardState(cardId, state)`, `loadCardState(cardId)`, `loadAllCardStates()`, `deleteCardState(cardId)`
- `saveStats(stats)`, `loadStats()`
- `saveAudioMuted(muted)`, `loadAudioMuted()`
- `clearAllData()`

### Web Implementation Details
**localStorage approach:**
- Card states: key prefix `ll.card.{cardId}` → JSON string
- Stats: key `ll.stats` → JSON string
- Audio muted: key `ll.audio_muted` → `"true"` / `"false"`
- `loadAllCardStates()`: iterate keys with prefix `ll.card.` — this is the only "scan" operation

**Why localStorage over IndexedDB:**
- Synchronous API matches MMKV's synchronous API perfectly (no async conversion needed)
- Service layer (`statsService.ts`, `fsrs.ts`, `cardSelector.ts`) all assume synchronous reads
- Data volume is small (~500 card states + 1 stats object = well under localStorage's 5MB limit)
- IndexedDB would require making the entire service layer async — huge refactor

**localStorage 5MB limit check:**
- ~500 cards × ~200 bytes/card = ~100KB for card states
- 1 stats object = ~500 bytes
- Total: ~100KB — well under the 5MB limit, even with growth

### File Structure
```
src/services/
  storage.ts          (existing — rename to storage.native.ts? or keep + add .web.ts)
  storage.web.ts      (new — localStorage implementation)
```

**Important:** Metro bundler resolves `.web.ts` automatically for web builds. The import `from './storage'` in other files doesn't change. No consumer code changes needed.

**Alternative considered:** Keep `storage.ts` as-is (it becomes the native file). Add `storage.web.ts` next to it. Metro picks `.web.ts` for web and `.ts` for native automatically. This is the cleanest approach — no renaming needed.

## 3. Service Worker & Offline Support

### Expo Does NOT Auto-Generate Service Workers
Expo static export produces plain HTML/JS/CSS. No service worker is included. We need to add one manually.

### Approach: Custom Service Worker
Create a minimal service worker that:
1. Caches the app shell (HTML, JS, CSS bundles) on install
2. Caches the content bundle (vocabulary data is baked into JS)
3. Uses cache-first strategy for static assets
4. Falls back to network for anything not cached

### Implementation
1. Create `public/sw.js` (Expo copies `public/` contents to `dist/` on export)
2. Register the service worker in the app entry point (or in a web-only component)
3. Service worker caches all JS/CSS bundles on install → app works offline after first load

### Service Worker Registration
Add registration in `app/_layout.tsx` (web-only):
```typescript
if (Platform.OS === 'web' && 'serviceWorker' in navigator) {
  navigator.serviceWorker.register('/sw.js');
}
```

### Cache Strategy
- **Install event:** Pre-cache app shell (index.html, JS/CSS bundles)
- **Fetch event:** Cache-first for static assets, network-first for API calls (none currently)
- **Update:** New service worker activates on next visit after deploy

## 4. Responsive Layout

### Current State
- All screens use `SafeAreaView` with `paddingHorizontal: 20`
- No max-width constraints — on desktop, content stretches to full window width
- Mobile-first styles work well on small screens

### Solution: Container Wrapper
Add a web-only container wrapper that:
- Centers content horizontally
- Limits max-width to ~480px (mobile phone width)
- Maintains full-height layout
- No changes to existing component styles

### Implementation Options

**Option A: Wrapper in `_layout.tsx` (recommended)**
```tsx
// Only on web: wrap Stack in a centered container
<View style={{ flex: 1, alignItems: 'center' }}>
  <View style={{ flex: 1, width: '100%', maxWidth: 480 }}>
    <Stack>...</Stack>
  </View>
</View>
```

**Option B: CSS in `public/` directory**
Add a `public/global.css` that targets the root element. Less React-native-ish.

Option A is preferred — stays within RN patterns, conditional on `Platform.OS === 'web'`.

## 5. Web Compatibility Issues

### Known Issues to Address

1. **`react-native-mmkv` import on web** — SOLVED by `storage.web.ts` platform file
2. **`SafeAreaView`** — `react-native-safe-area-context` provides web support (insets are 0 on web, which is correct)
3. **`react-native-paper`** — Fully supports web via `react-native-web` (MD3 components render as HTML)
4. **`expo-router`** — Works on web out of the box with file-based routing
5. **`ts-fsrs`** — Pure TypeScript, no native dependencies, works everywhere
6. **`fuse.js`** — Pure JavaScript, works everywhere
7. **Deep linking (`useDeepLink` hook)** — Uses `expo-linking` which has web support. URL scheme won't work on web (no `lingolock://`) but the home screen "Start Practice" button navigates directly via router, so this is fine.
8. **`react-native-screens`** — Has web support, but may need `react-native-screens/native` import check
9. **`react-native-vector-icons`** — Needs web font loading configuration

### Low Risk
Most dependencies are web-compatible. The only significant change is the storage adapter.
