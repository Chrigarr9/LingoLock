# Phase 02.3: Audio Generation Pipeline — Context

**Phase Goal:** Extend Spanish content pipeline with AI text-to-speech generation for sentence audio, enabling audio playback in vocabulary cards

**Scope Boundary:** This phase is ONLY about content generation (Pass 6 in pipeline). Playback UI already exists (CARD-10 complete). Any playback features (speed control, repeat, etc.) are OUT OF SCOPE.

---

## 1. TTS Provider & Voice Selection

### Multi-Provider Support (Like Images)
- **Primary provider:** Google Cloud TTS (API key already available)
- **Secondary provider:** OpenAI TTS (if API key provided and selected in config)
- **Config-driven:** Provider selection via deck config YAML (same pattern as image generation)
- **Fallback:** If configured provider fails, don't fall back to another - fail gracefully like images

### Voice Characteristics
- **Language:** Spanish (from config)
- **Accent:** Neutral Spanish, clearly understandable, no heavy regional accents
- **Gender:** Male voice
- **Style:** Natural, educational tone suitable for language learning

### Audio Quality & File Size
- **Format:** MP3 (universal compatibility - web and native)
- **Bitrate:** 64kbps (balanced quality/size tradeoff)
- **Optimization:** Keep file size as small as possible while sounding normal (not high fidelity needed)
- **Expected size:** ~15KB per sentence, ~5-10MB per deck

### Generation Scope
- **Coverage:** Generate audio for ALL sentences in the content
- **No selective generation:** Every sentence gets audio if TTS succeeds

---

## 2. Pipeline Integration & Error Handling

### Pipeline Position
- **Pass 6:** Audio generation runs AFTER image generation (Pass 5)
- **Sequential:** Not parallel with images - run as separate final pass
- **Integration point:** Add to `run_all.py` after image generation block

### Error Handling Strategy
- **API failure:** Skip that sentence and continue (same as image generation)
- **No retries:** Don't retry failed sentences automatically
- **Card behavior:** If audio generation fails for a sentence, card gets no audio (audioUrl remains undefined/null)
- **Graceful degradation:** App already handles missing audio (CARD-10 designed for optional audio)

### Optional/Skippable Execution
- **Same as image generation:**
  - If no TTS API key configured → skip Pass 6 entirely (with warning message)
  - Command-line flag support: `--skip-audio` to skip audio generation even if key present
  - Config flag: `audio_generation.enabled: false` to disable audio in deck config
- **Never block pipeline:** Audio generation failures don't fail entire pipeline run

### Incremental Updates (Manifest-Based)
- **Audio manifest:** Track generated audio files (like image manifest)
- **Content hash:** Check if sentence text changed since last generation
- **Regeneration logic:**
  - Audio file exists + sentence unchanged → skip
  - Audio file missing OR sentence changed → generate
- **Cost optimization:** Only pay for new/changed sentences on re-runs

---

## 3. File Organization & Bundle Integration

### File Storage Structure
- **Mirror image organization:** Same pattern as `output/{deck-id}/images/`
- **Audio directory:** `output/{deck-id}/audio/`
- **File naming:** Match sentence IDs (e.g., `hola-ch01-s01.mp3`)
- **Flat structure:** All audio files in single audio directory for simplicity

### Version Control
- **Commit audio files:** Audio is content, should be versioned in git
- **File size acceptable:** 5-10MB per deck is manageable in git
- **Same as images:** Images are committed, audio follows same pattern

### Bundle Integration
- **Existing pattern:** Follow how images are included in bundle.ts
- **ClozeCard type:** Add optional `audioUrl` field (likely already exists from CARD-10)
- **Asset imports:** Audio files imported/required like images in build-content.ts
- **Single directory reference:** One audio directory, cards link to files within it

### Cross-Platform Compatibility
- **Single format:** MP3 works on web and native (no platform-specific variants)
- **No separate formats:** Don't generate AAC/OGG variants
- **Universal playback:** expo-av handles MP3 on both platforms

---

## 4. Cost, Caching & Development Experience

### Development Workflow
- **No special dev mode:** Use same TTS provider in dev and production
- **Chapter-based testing:** During development, generate only chapters 1-2 to limit costs
- **Full generation:** Once satisfied with chapters 1-2, run full pipeline for all chapters
- **Cost control:** Manual chapter selection is sufficient (`--chapters 1-2` flag)

### Cost Estimation
- **No dry-run needed:** Don't implement cost estimation feature
- **User manages costs:** User controls via chapter selection and incremental generation

### Multi-Language Support (Future)
- **Design for configurability:** Language should be driven by deck config
- **Automatic adaptation:** TTS prompts/parameters adapt based on config.story.target_language
- **No hardcoding:** Don't hardcode "Spanish" - read from config
- **Voice selection:** Provider and voice parameters should be config-driven for future language packs

### Out of Scope for Phase 02.3
- **Playback speed control:** UI feature for future phase
- **Audio speed variants:** Don't generate slow/normal/fast versions
- **Playback UI changes:** CARD-10 complete, no changes to audio button/player
- **Audio quality settings:** Single quality level (balanced) for all content
