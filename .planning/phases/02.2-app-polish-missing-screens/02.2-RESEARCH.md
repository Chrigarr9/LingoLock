## Summary

Phase 02.2 is a pure React Native / Expo Router development phase — no new native libraries needed. The entire stack is already installed and working. Three new screens (Vocabulary browser, Stats detail, Settings) follow the established Expo Router Stack pattern: add a `<Stack.Screen>` entry in `_layout.tsx` and create the corresponding file in `app/`. The challenge screen needs a significant refactor from fixed-count to continuous-flow sessions, which requires new logic in `cardSelector.ts` (daily new-word tracking) and changes to `challenge.tsx` (session termination condition).

The daily new-word cap is the most architecturally interesting change. It requires tracking how many new cards have been introduced today — a new field in MMKV storage, following the same pattern already used for streak/abort tracking. The cap logic belongs in `cardSelector.ts`'s `buildSession()` which currently has no concept of a daily budget.

The vocabulary browser and stats detail screens are primarily data-display work. All the underlying data already exists: `CHAPTERS`, `getChapterCards()`, `getChapterMastery()`, `getCardsDueCount()`, `loadCardState()`, `isCardMastered()`. No new service methods need to be invented — only composed. The cross-navigation (stats chapter row → vocabulary filtered to that chapter) is a router param on the existing vocabulary route.

**Primary recommendation:** No new dependencies needed. Build in this order: (1) settings + new-per-day storage, (2) cardSelector daily cap, (3) continuous session flow, (4) new screens (vocabulary, stats), (5) home polish (greeting, empty state, hide tutorial on web).

```typescript
// In storage.ts — add two new functions
const NEW_WORDS_PER_DAY_KEY = 'new_words_per_day';
const NEW_WORDS_TODAY_KEY = 'new_words_today';
const NEW_WORDS_TODAY_DATE_KEY = 'new_words_today_date';

export function loadNewWordsPerDay(): number {
  return statsStorage.getNumber(NEW_WORDS_PER_DAY_KEY) ?? 20;
}
export function saveNewWordsPerDay(count: number): void {
  statsStorage.set(NEW_WORDS_PER_DAY_KEY, count);
}
export function loadNewWordsIntroducedToday(): number {
  const date = statsStorage.getString(NEW_WORDS_TODAY_DATE_KEY);
  const today = new Date().toISOString().slice(0, 10);
  if (date !== today) return 0; // reset on new day
  return statsStorage.getNumber(NEW_WORDS_TODAY_KEY) ?? 0;
}
export function recordNewWordsIntroduced(count: number): void {
  const today = new Date().toISOString().slice(0, 10);
  statsStorage.set(NEW_WORDS_TODAY_DATE_KEY, today);
  const current = loadNewWordsIntroducedToday();
  statsStorage.set(NEW_WORDS_TODAY_KEY, current + count);
}
```

### Continuous Session Architecture

The major structural change in `challenge.tsx`. Current model: `originalCardCount.current` is fixed at session start; `advanceToNext` completes when `nextIndex >= originalCardCount.current`.

**New model:**

`buildSession()` in `cardSelector.ts` must change signature to accept the daily new-word budget:

```typescript
export function buildSession(dailyNewWordBudget: number, _sourceApp?: string): SessionCard[]
```

The session is no longer "N cards" — it's "all due reviews + up to `dailyNewWordBudget` new words not yet introduced today". `buildSession` returns this dynamic count.

Session completion condition in `challenge.tsx` becomes: queue is exhausted (no more cards after current one). The queue IS the complete session — when it runs out, show celebration.

**Daily new-word tracking in buildSession:**
```typescript
// In buildSession — after collecting newCards array:
const newWordsIntroducedToday = loadNewWordsIntroducedToday();
const remainingBudget = Math.max(0, dailyNewWordBudget - newWordsIntroducedToday);
const selectedNew = newCards.slice(0, remainingBudget);
// After building selected array, record how many new cards we're about to introduce:
recordNewWordsIntroduced(selectedNew.length);
```

**challenge.tsx refactor — key points:**
1. Remove `cardCount` param parsing — session size is dynamic, determined by `buildSession`
2. `originalCardCount.current` still useful for progress bar denominator — set to `session.length` at init
3. Session end: `nextIndex >= queue.length` (queue is authoritative, no wrong-answer re-insertions extend past original count for progress — same as today via `originalCardCount.current`)
4. Actually: wrong-answer re-insertions still work the same way (they extend queue but NOT `originalCardCount.current`). The new end condition is when `currentIndex` reaches `originalCardCount.current` (the original size, not the re-inserted-extended queue)
5. `params.count` still exists for deep-link compatibility but voluntary practice ignores it

**CTA button on home screen changes:**
```typescript
// Old:

// New — count is irrelevant for continuous practice, but keep type for stats:

```

### Anti-Patterns to Avoid

- **Do not import `bundle.ts` CHAPTERS in screens directly for mastery computation** — `getChapterMastery()` already does this. Call the service function.
- **Do not call `loadCardState()` in a render loop without memoization** — vocabulary list may have 50+ cards. Use `useMemo` to compute all mastery states once, not per-render.
- **Do not put new-word tracking inside challenge.tsx** — belongs in `cardSelector.ts`/`storage.ts`. The screen should not know about daily budgets.
- **Do not forget the web storage adapter** — every new function in `storage.ts` must have a parallel implementation in `storage.web.ts`. Missing this breaks PWA.
- **Do not animate ProgressDots for continuous session** — `ProgressDots` uses `originalCardCount.current` as denominator. This is still correct. Do not rethink it.

## Common Pitfalls

### Pitfall 1: Web Storage Adapter Drift

**What goes wrong:** New storage functions added to `storage.ts` but forgotten in `storage.web.ts`. PWA works fine on native, breaks silently on web (no error thrown, just returns undefined).

**Why it happens:** Two files must stay in sync manually. Easy to forget the web adapter when focused on native logic.

**How to avoid:** For every new function in `storage.ts`, immediately write the parallel implementation in `storage.web.ts` in the same commit. The MMKV API maps directly: `getNumber()` → `localStorage.getItem()` + `parseInt`, `set(key, number)` → `localStorage.setItem(key, String(value))`.

**Warning signs:** Tests pass but PWA at vercel shows wrong defaults (always 20, never persists changes).

### Pitfall 2: Daily Budget Double-Count on Session Restart

**What goes wrong:** `recordNewWordsIntroduced()` called every time `buildSession()` runs. If the user exits and re-enters practice, new words are double-counted against the daily budget.

**Why it happens:** `buildSession` is called on challenge screen mount. If user exits mid-session and returns, `buildSession` runs again and adds to the already-recorded count even though those cards were already introduced.

**How to avoid:** Track which specific card IDs were introduced today (or simply don't record in `buildSession` — record in `updateStatsAfterSession` instead, based on actual cards presented). Simpler approach: record in `buildSession` only the DELTA between `loadNewWordsIntroducedToday()` at start and the new count — and only once per session, not per call.

Alternative: make `buildSession` return the session without side effects. Record new-word introductions in `updateStatsAfterSession` by counting how many cards in the completed session had null prior state.

**Recommendation:** Count new words introduced in `challenge.tsx` at session completion (tracking which cards were new at session start), then call a new `recordNewWordsIntroduced(count)` only in `updateStatsAfterSession`. Keeps `buildSession` pure/idempotent.

### Pitfall 3: SectionList + Search Filter Re-render Performance

**What goes wrong:** Search state updates on every keystroke, recomputing `sections` array from scratch including `loadCardState()` calls for all 50+ cards.

**Why it happens:** `loadCardState` is synchronous MMKV reads. On each keystroke a `useState` change triggers render, which recomputes sections.

**How to avoid:** Use `useMemo` for mastery status computation (keyed on a version counter or focus count, not the search string). Compute mastery status once on focus, then filter/search is pure JS object manipulation.

```typescript
const masteryMap = useMemo(() => {
  const map: Record<string, 'new' | 'learning' | 'mastered'> = {};
  for (const ch of CHAPTERS) {
    for (const card of ch.cards) {
      const state = loadCardState(card.id);
      if (state === null) map[card.id] = 'new';
      else if (isCardMastered(state)) map[card.id] = 'mastered';
      else map[card.id] = 'learning';
    }
  }
  return map;
}, [focusKey]); // focusKey increments on useFocusEffect
```

### Pitfall 4: challenge.tsx "count" param Conflict

**What goes wrong:** `challenge.tsx` currently reads `params.count` and uses it as the session size. Home screen currently passes `count: '5'`. If we change home screen to pass `count: '0'` for continuous practice but existing deep links still pass a count, behavior becomes inconsistent.

**Why it happens:** The count param was designed for forced sessions (unlock/app_open) where a fixed count is meaningful. Voluntary practice shouldn't use it.

**How to avoid:** Add a session mode to the route params: `mode: 'continuous' | 'fixed'`. When `mode === 'continuous'`, ignore `count` and use `buildSession(dailyNewWordBudget)`. When `mode === 'fixed'`, keep existing fixed-count behavior for forward-compat with future forced sessions. Default to `'continuous'` when param is absent.

### Pitfall 5: Vocabulary Detail View — ClozeCard Without Audio

**What goes wrong:** Reusing `ClozeCardDisplay` for the vocabulary detail view triggers audio playback and shows answer-state UI that doesn't make sense outside a challenge session.

**Why it happens:** `ClozeCardDisplay` is designed for challenge flow: it plays audio when `showAnswer=true` and shows correct/incorrect color states.

**How to avoid:** Don't reuse `ClozeCardDisplay` directly in the vocabulary detail. Build a simpler `VocabularyDetailCard` that shows the sentence, image, German translation, FSRS status, and chapter — without the challenge mechanics. It can reuse the image rendering and sentence display logic but not the audio/answer reveal.

## Open Questions

1. **Daily new-word tracking — when to record?**
   - What we know: `buildSession` must respect the daily budget; the count must persist across sessions
   - What's unclear: best place to record the introduction — in `buildSession` (risks double-count on restart) or in `challenge.tsx` at session end (requires tracking which cards were new at start)
   - Recommendation: Track which cards were new at session start via a `isNewCard` flag on `SessionCard`, then count + record in `updateStatsAfterSession`. Keeps `buildSession` pure and avoids double-counting.

2. **Vocabulary detail view — separate route or modal?**
   - What we know: Stack pattern gives free back navigation; `app/vocabulary/[id].tsx` requires nested routing which Expo Router supports
   - What's unclear: whether nested route `vocabulary/[id]` needs explicit Stack.Screen registration in `_layout.tsx`
   - Recommendation: Register explicitly to control title/back behavior. Alternatively, use a simple state-based in-screen detail panel (show/hide) to avoid routing complexity — simpler for this phase.

3. **Settings new-per-day control — number input or +/- stepper?**
   - What we know: This is at Claude's discretion per CONTEXT.md
   - What's unclear: `react-native-paper` does not include a stepper component out of the box
   - Recommendation: Use two `IconButton` (+/-) with a `Text` display in between. Simple, no extra library, consistent with Paper design language.

4. **challenge.tsx continuous mode — back button behavior**
   - What we know: User can go back at any time (voluntary exit, not counted as abort per CONTEXT.md)
   - What's unclear: Should `recordAbort` be called for voluntary Practice exits? Current code calls it for all forced sessions.
   - Recommendation: Only call `recordAbort` when `params.type === 'unlock'` (forced). Voluntary `'app_open'` practice exits from the new continuous flow should NOT call `recordAbort`. The current logic `isForced && !isComplete` should be narrowed to only `params.type === 'unlock'`.
