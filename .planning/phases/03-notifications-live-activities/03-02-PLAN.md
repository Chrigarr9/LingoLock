  <files>package.json, app.json, src/services/widgetService.ts, src/services/widgetService.web.ts</files>
  <action>
    Install expo-widgets:
    ```
    npx expo install expo-widgets
    ```

    Add expo-widgets to app.json plugins if required by the library.

    Create `src/services/widgetService.ts`:

    1. **WidgetCardData** interface:
       ```typescript
       export interface WidgetCardData {
         cardId: string;
         sentence: string;       // Cloze sentence with _____ gap
         germanHint: string;     // German hint word
         correctAnswer: string;  // wordInContext
         answerType: 'mc2' | 'mc4' | 'text';  // Determines widget interaction mode
         choices?: string[];     // MC choices for mc2/mc4 cards
         imageUri?: string;      // Optional card image
         cardsLeft: number;      // Total cards remaining in session
         streakCount: number;    // Current streak
       }
       ```

       The answerType field determines widget behavior:
       - `mc2` / `mc4`: Widget shows A/B or A/B/C/D answer buttons directly (iOS 17+ Button support)
       - `text`: Widget shows tap-to-open deep link (iOS widgets cannot have text input fields)

    2. **getWidgetCardData()** function:
       - Import getDueCards logic from cardSelector (reuse the due-card scanning logic)
       - Scan all CHAPTERS for cards with stored state where isDue() is true (same as buildSession's due card logic)
       - Per user decision on widget content filtering:
         - If images supported: show both new + repetition cards (full FSRS queue)
         - If images NOT supported: only repetition cards (seen at least once = has CardState)
         - Start with repetition-only for Lock Screen (conservative approach)
       - Get streak from statsService.getStreak()
       - Return WidgetCardData for the first due card, or null if no cards due
       - Also return cardsLeft count

    3. **updateWidgetData()** function:
       - Call getWidgetCardData() to get current card
       - If expo-widgets provides an updateTimeline/updateSnapshot API, call it with the card data
       - This function should be called after every answer (in-app, notification, or widget) to keep widget fresh
       - If expo-widgets API not available at runtime (web, or pre-SDK 55), silently no-op

    4. **processWidgetAnswer(cardId: string, selectedChoice: string)** function:
       - Called when user taps an MC answer button on the widget
       - Load ClozeCard from content bundle using cardId
       - Load or create CardState for the cardId
       - Validate answer: compare selectedChoice to correctAnswer (wordInContext)
       - Call scheduleReview(cardState, isCorrect) from fsrs.ts
       - Call saveCardState() to persist
       - Call updateStatsAfterSession(isCorrect ? 1 : 0, 1, 'widget') from statsService
       - Call updateWidgetData() to refresh widget with next card
       - Returns { isCorrect: boolean, correctAnswer: string } for widget UI feedback

    5. **clearWidgetData()** function:
       - Clear widget state when user completes all cards
       - Widget should show empty state

    Create `src/services/widgetService.web.ts` with no-op stubs:
    - `getWidgetCardData()` returns null
    - `updateWidgetData()` returns void
    - `processWidgetAnswer()` returns { isCorrect: false, correctAnswer: '' }
    - `clearWidgetData()` returns void
    - Export same WidgetCardData type

    IMPORTANT: expo-widgets is very new (SDK 55, January 2026). The exact API surface may differ from research examples. If the createWidget/updateTimeline APIs are not available in the installed version:
    - Still create the widgetService with the data preparation logic (getWidgetCardData, processWidgetAnswer)
    - Create the widget component file as a placeholder with TODO comments for the actual widget rendering
    - The data layer and answer processing are the critical pieces -- widget rendering can be adjusted when testing on device
