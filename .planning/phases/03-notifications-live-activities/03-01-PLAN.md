<task type="auto">
  <name>Task 2: Create notification service with categories, permissions, and handler</name>
  <files>src/services/notificationService.ts, src/services/notificationService.web.ts, app/_layout.tsx</files>
  <action>
    Create `src/services/notificationService.ts` with:

    1. **Foreground notification handler** -- configure `Notifications.setNotificationHandler` to show alerts without sound (shouldShowBanner: true, shouldShowList: true, shouldPlaySound: false, shouldSetBadge: false). Call this at module top-level so it's active immediately.

    2. **registerNotificationCategories()** function -- register a `vocabulary-challenge` category with these actions:
       - `answer-text`: buttonTitle "Type Answer", with textInput (submitButtonTitle: "Submit", placeholder: "Enter answer..."), opensAppToForeground: false
       - `answer-a`: buttonTitle "A", opensAppToForeground: false
       - `answer-b`: buttonTitle "B", opensAppToForeground: false
       - `answer-c`: buttonTitle "C", opensAppToForeground: false
       Note: iOS shows up to 4 actions. For MC2 cards, only answer-a and answer-b will have content. For MC4, all four will have content. The button titles will be overridden per-notification with actual choice text.

       Also register a `vocabulary-mc` category for multiple choice only (no text input action):
       - `answer-a` through `answer-d` with opensAppToForeground: false

       NOTE: iOS notification categories have action identifiers that remain fixed. The actual button labels are set per action on the category, not per notification. For MC mode, set category with answer-a through answer-d. The action button titles will show the choices. Since iOS doesn't let you change action titles per notification, use a single category with text input as primary action. MC answers will be handled by the text input -- user types the letter (A/B/C/D) or the full answer.

       REVISED APPROACH: Use a single category `vocabulary-challenge` with one text input action:
       - `answer-text`: buttonTitle "Answer", textInput (submitButtonTitle: "Submit", placeholder: "Type your answer..."), opensAppToForeground: false

       This works for both text and MC modes. For MC notifications, the body will show the choices as "A) option1  B) option2 ..." and the user types their answer. This is simpler and avoids the iOS limitation of fixed action titles per category.

    3. **requestNotificationPermissions()** function -- async function that:
       - Calls `Notifications.getPermissionsAsync()` to check current status
       - If 'granted', return true
       - If 'denied', show Alert directing user to Settings (use `Linking.openSettings()`)
       - If 'undetermined', call `Notifications.requestPermissionsAsync()` and return status === 'granted'
       - Returns boolean indicating whether permissions are granted

    4. **setupNotifications()** function -- orchestrator that:
       - Checks Platform.OS !== 'web' (skip on web)
       - Calls registerNotificationCategories()
       - Registers `addNotificationResponseReceivedListener` that will be wired to a callback. For now, just log the response. Plan 03 will wire the actual answer processing.
       - Export the response listener subscription so _layout.tsx can clean up
       - Returns a cleanup function that removes the listener

    5. **Export types:**
       - `NotificationData` interface: { cardId: string, correctAnswer: string, choices?: string[], answerType: 'text' | 'mc2' | 'mc4', deliveryTime: number }

    Create `src/services/notificationService.web.ts` with no-op stubs:
    - `setupNotifications()` returns empty cleanup function
    - `requestNotificationPermissions()` returns false
    - `registerNotificationCategories()` returns void
    - Export same `NotificationData` type

    Update `app/_layout.tsx`:
    - Import `setupNotifications` from `../src/services/notificationService`
    - In the existing useEffect (where service worker registration happens), add a call to `setupNotifications()` for native platforms (Platform.OS !== 'web')
    - Store cleanup function and return it from useEffect
    - Keep existing deep link handling and service worker registration unchanged
