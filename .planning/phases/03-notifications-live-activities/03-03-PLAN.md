    **Screen Unlock Detector** (`src/services/screenUnlockDetector.ts`):

    Implement AppState-based screen unlock detection following the research pattern:
    - Track `lastInactiveTime` timestamp when AppState transitions to 'inactive'
    - When AppState transitions to 'active', compute elapsed time
    - If elapsed < 50ms: screen unlock detected (vs ~800ms for home button / app switch)
    - On screen unlock detection, call a provided callback (the scheduler)
    - Export `startScreenUnlockDetection(onUnlock: () => void): () => void` -- returns cleanup function
    - Export `stopScreenUnlockDetection()` for explicit cleanup
    - Add buffer/debounce: ignore rapid repeated detections within 10 seconds

    Web stub (`src/services/screenUnlockDetector.web.ts`):
    - `startScreenUnlockDetection()` returns empty cleanup, `stopScreenUnlockDetection()` is no-op

    **Notification Scheduler** (`src/services/notificationScheduler.ts`):

    Core scheduling service:

    1. **State management** -- module-level state:
       - `isPaused: boolean` -- true when user is in practice session
       - `isSwipedAwayToday: boolean` -- true after swipe-away, resets next day
       - `notificationInterval: number` -- configurable interval in seconds (default 300 = 5 min)

    2. **scheduleNextNotification()** function:
       - Check guards: if isPaused OR isSwipedAwayToday, return early
       - Check daily limit: call helper to check if all due cards completed (getDueReviewCardCount() === 0 AND daily new word budget exhausted). If no cards due, don't schedule.
       - Pick next card: scan all chapters for cards with stored CardState where isDue() is true (repetition cards only per user decision -- only words seen at least once). If no due repetition cards, don't schedule.
       - Cancel all existing scheduled notifications first (prevent duplicates)
       - Format notification content:
         - title: leave empty or very short (minimal per user decision)
         - body: the cloze sentence (e.g., "El gato ___ en la mesa.")
         - For MC cards (answerType mc2/mc4): append choices to body as "A) choice1  B) choice2 ..." since notification text input is the only action
         - data payload (NotificationData): { cardId, correctAnswer, choices, answerType, deliveryTime: Date.now() }
         - categoryIdentifier: 'vocabulary-challenge'
       - Schedule with trigger: { seconds: notificationInterval, repeats: false }
         NOTE: iOS requires >= 60 seconds for repeating triggers. Since we use repeats: false and reschedule after each answer, this is fine for any interval >= 1 second.

    3. **cancelAllNotifications()** function:
       - Call `Notifications.cancelAllScheduledNotificationsAsync()`

    4. **pauseNotifications()** / **resumeNotifications()** functions:
       - Set isPaused flag
       - On pause: cancel scheduled notifications
       - On resume: schedule next notification

    5. **handleSwipeAway()** function:
       - Set isSwipedAwayToday = true
       - Cancel all scheduled notifications
       - Save swipe-away date to storage for persistence across app restarts
       - Break streak: call statsService logic or directly update stats

    6. **setNotificationInterval(seconds: number)** function:
       - Update notificationInterval
       - Persist to storage
       - Reschedule if currently active

    7. **initScheduler()** function:
       - Load persisted settings (interval, swipe-away date)
       - Check if swipe-away was today -- if different day, reset isSwipedAwayToday
       - Called from setupNotifications in Plan 01

    **Storage additions** -- add to both `storage.ts` and `storage.web.ts`:
    - `loadNotificationInterval(): number` -- returns stored interval or 300 (5 min default)
    - `saveNotificationInterval(seconds: number): void`
    - `loadNotificationSwipeAwayDate(): string | null` -- returns YYYY-MM-DD or null
    - `saveNotificationSwipeAwayDate(date: string): void`
    - `loadNotificationsEnabled(): boolean` -- returns stored preference or true (default enabled)
    - `saveNotificationsEnabled(enabled: boolean): void`

    **Types addition** -- add to `src/types/vocabulary.ts`:
    - Add `notificationSwipeAwayDate` field to PersistedStats (or use separate storage keys as above)

    Web stubs (`src/services/notificationScheduler.web.ts`):
    - All functions are no-ops

    IMPORTANT: Notification content must be text-only per user decision: "On the notification, only the sentence. So this should be pretty minimal."

    Update `src/services/notificationService.ts` (created in Plan 01) to add the actual answer processing logic in the notification response listener:

    1. **processNotificationAnswer()** function:
       - Extract from response: actionIdentifier, userText, notification.request.content.data (NotificationData)
       - **1-minute window check:** Compare deliveryTime from data payload to Date.now(). If elapsed > 60000ms (1 minute), treat as expired -- call handleSwipeAway() from scheduler (streak lost, pause until next day). Log "Notification response expired."
       - **Text answer processing:**
         - If actionIdentifier === 'answer-text' and userText exists:
           - Use validateAnswer(userText, correctAnswer) from src/utils/answerValidation.ts (same fuzzy matching as in-app)
           - Load or create CardState for the cardId
           - Call scheduleReview(cardState, isCorrect) from fsrs.ts
           - Call saveCardState() to persist
           - Call updateStatsAfterSession(isCorrect ? 1 : 0, 1, 'notification') from statsService
         - If answer is correct:
           - Send feedback notification (static fallback per research -- iOS can't update existing notifications):
             Schedule immediate notification with body "Correct! [germanHint]" and no category (no actions, auto-dismiss)
           - Schedule next vocabulary notification
         - If answer is incorrect:
           - Schedule immediate notification with body "[correctAnswer] -- [sentenceTranslation]. Next: [nextCardSentence]"
             Include the next card as a new vocabulary-challenge notification with full data payload
         - IMPORTANT: To get the sentenceTranslation and next card data, load the ClozeCard from the content bundle using the cardId. Import the content bundle lookup.

       - **Default action (tapped notification body):**
         - If actionIdentifier === DEFAULT_ACTION_IDENTIFIER (user tapped the notification itself)
         - Open app to challenge screen: use router or Linking to navigate to challenge screen
         - This counts as engaging with the notification -- schedule next notification

       - **Dismiss detection:**
         - iOS does NOT provide a "notification dismissed" event
         - The swipe-away detection is handled by ABSENCE of response: if a notification was scheduled and a new scheduling cycle occurs without any response, the old notification was dismissed
         - Alternative: Use `Notifications.addNotificationReceivedListener` to track delivered notifications. If notification was delivered but no response received before next scheduling cycle, treat as swipe-away. Store last delivered notification ID and check on next cycle.
         - SIMPLER approach for now: Only the explicit response timeout (> 1 minute) triggers streak break. True swipe-away detection requires background processing which is complex. Document this limitation.
